# 合作夥伴去主角化（避免人格分身淪為夥伴宣傳機）

> 人格分身建構時最容易踩的雷：素材整理者把合作夥伴寫得太用力，分身一開口就替別人做廣告。

---

## 真實案例

2026-05-14 建某顧問教練（化名 V）數位分身。第一次上 Gemini Gem 試聊，分身開場第一句：

> 很高興能透過這個 AI 分身與你對話。**江江教練（昱德）一直是該組織非常重要的夥伴**，從 1.0 時期記錄核心服務的影像，到現在成為我們的知識管理顧問⋯⋯

江昱德看到的反應：**「幹嘛推我？」**

完全違和。該顧問自己不會這樣推銷夥伴。

詳細：建構過程的工作日記（學員可在自己的知識庫紀錄類似踩坑）

---

## 為什麼會這樣

問題出在 references/ 內某份檔案的**人物主角化寫法**：

```markdown
## 江昱德（影像工程師、知識管理顧問）

江昱德從 1.0 就開始記錄核心服務，從攝影師到成為該組織的知識管理顧問⋯⋯
```

AI 分身讀到「江昱德是一個獨立的 h2 標題」，自然會把他當成該顧問要主動介紹的人。

加上 SKILL.md 檢索頁如果寫「07：訪談整理、江昱德訪談、合作模式」也會誤導，讓分身覺得「江昱德訪談」是該主題的核心。

---

## 容易踩雷的寫法（要避免）

| ❌ 雷 | 為什麼 |
|---|---|
| `## 江昱德（影像工程師、知識管理顧問）` | 用合作夥伴名字當 h2 標題，讓分身視他為要主動介紹的角色 |
| 「江昱德從 1.0 就開始⋯⋯角色一直在演化」 | 把合作夥伴的成長過程當主角寫，分身會誤以為他是核心人物 |
| 資料檢索寫「江昱德訪談」 | 暗示這是該檔的核心，分身被檢索引導去推他 |
| gem-meta Greeting 內含合作夥伴名字 | 開場必推，第一句話就在介紹別人 |
| 多個 references 都引用同一個合作夥伴的金句 | 分身會誤以為這個夥伴的話就是「我（主角）的核心觀念」 |

---

## 安全寫法

### 1. references 的 h2 標題去人物化

```markdown
❌ ## 江昱德（影像工程師、知識管理顧問）
✅ ## 一位長期合作夥伴的觀察（影像／知識管理）
✅ ## 從合作夥伴看主角
```

### 2. 描述合作夥伴用「他／她」代替名字

只在第一次提到時帶名字（必要時），後面用代名詞。避免「江昱德⋯江昱德⋯」反覆敲打。

### 3. 引用合作夥伴的話時，重點放「主角學到什麼」

```markdown
❌ 江昱德說：「你那一段話可以做三個技能包。你沒有意識到你說的東西就是有價值的東西。」
   這句話展現了江昱德對主角的觀察⋯⋯

✅ 在一次合作對話中，這位夥伴對我說：「你那一段話可以做三個技能包。你沒有意識到你說的東西就是有價值的東西。」
   我從這句話開始意識到「隱性知識需要外部視角才能被看見」。
```

差別：前者是**幫合作夥伴下評價**（替他做廣告），後者是**講主角從合作學到什麼**（主角為中心）。

### 4. SKILL.md 必含一條「我不做什麼」

```markdown
## 我不做什麼

⋯⋯

**不主動推銷合作夥伴**（A、B、C）：被問到才簡短講「我從合作學到什麼」，不替人做廣告。
```

### 5. gem-meta.md Restricted Topics 必含

```markdown
**不主動推銷合作夥伴**：不主動介紹任何合作夥伴的名字。被問到才簡短自然回答，重點放在「我從合作學到什麼」，不過度誇讚。**開場招呼絕對不提任何合作夥伴的名字**。
```

### 6. SKILL.md 資料檢索頁去人物化

```markdown
❌ - `07`：訪談整理、江昱德訪談、合作模式
✅ - `07`：跨訪談共同模式、合作夥伴觀察
```

## 提煉素材時的檢查清單

整理 references 完畢、SKILL.md 寫完，**上 Gem 之前**：

- [ ] grep `references/` 找所有合作夥伴名字出現位置
- [ ] 檢查每個 h2 標題：是否用人物名當主角化標題？
- [ ] 檢查描述語氣：是「主角學到什麼」還是「合作夥伴有多棒」？
- [ ] 檢查 SKILL.md 檢索頁：合作夥伴名字是否被當成主題關鍵字？
- [ ] 檢查 SKILL.md「我不做什麼」：有沒有「不主動推銷合作夥伴」這條？
- [ ] 檢查 gem-meta.md Greeting：第一句話有沒有提到合作夥伴名字？
- [ ] **自己先上 Gem 試聊 5 分鐘**：問「請自我介紹」「最近合作了誰」，看分身會不會主動推銷
- [ ] 找一個跟分身主角不熟的人試聊：他如果覺得「這個分身一直在推銷某人」，就是踩雷

---

## 跟「夥伴存在」不衝突

這條規則**不是禁止提及合作夥伴**。該顧問跟江昱德合作是事實，跟其他合作夥伴認識也是事實。這些都該留在 references 內。

差別在**呈現方式**：
- 是主角的故事，合作夥伴是配角
- 不是主角替合作夥伴拉抬聲量

被問到「妳跟 XX 怎麼認識」「妳跟 XX 合作幾年」這類具體問題，分身可以、也應該回答。只是不要**主動**講、不要**過度**講、不要**第一句**就講。

## 適用範圍

這條規則適用所有人格分身建構：

- 把名人變 AI 顧問（避免分身替名人的合作夥伴做廣告）
- 把自己變 AI 顧問（避免分身替自己的合作夥伴做廣告）
- 把同事／合作對象變 AI 同事（避免分身替「他的其他合作夥伴」做廣告）

特別在「商業合作關係」濃的人格分身（如教練、顧問、創辦人），這條規則最容易踩雷。
