## 案例 1：基金會講座回饋系統

### 背景

一位文教基金會主管在參加完活動後找上我：

> 「江江教練，我知道您是 AI 應用規劃師，也很會整理資料。是否可以協助我們設計一個更簡單、更自動化的系統，來收集講座後聽眾的回饋？」

### 第一反應（差點犯的錯）

我當下立刻回答：「好呀！我覺得這很適合用官方 LINE 收集，還能用 AI 協助統整。我回去跟工程師討論一下，再給你一份方案說明。」

### 翻轉的瞬間

回家路上我開始想：

- 講座結束後，聽眾真的會有想法嗎？
- 他們真的想分享嗎？
- 他們實際上都怎麼分享？分享給誰？

我意識到：在規劃方案之前，必須先弄清楚真實情況跟需求。

於是我把整套需求放進真實需求調查法的框架，重新設計訪談策略。

### 訪談策略設計

**目標：** 不推銷任何解決方案，而是深入挖掘經理觀察到的「現象」與背後的「動機」。

**核心心態：** 我是來了解問題的，而不是來推銷方案的。

**開場白：**
> 「經理，關於上次我們提到收集聽眾回饋的想法，我回去深入思考後，覺得在決定『用什麼工具』之前，我們可能需要先更深入地了解聽眾們的真實情況與想法。這樣才能確保我們做出來的東西，是他們真的會用、也真的能解決問題的。所以今天想跟您請教幾個問題，來釐清整個輪廓。」

### 訪談問題設計（問經理）

**挖掘動機與問題根源（V）**
- 當初是什麼具體事件，讓您覺得「需要系統化收集回饋」這件事很重要？
- 如果一直「沒有」這些回饋，對基金會最大的困擾或遺憾是什麼？

**了解現況與既有行為（P）**
- 目前，聽眾們最常透過哪些方式分享心得？
- 過去收集到的回饋都如何被應用？

**定義成功的樣貌（J）**
- 假設三個月後這個機制運作得非常好，那會是什麼樣子？
- 我們如何運用這些回饋來達成目標？

### 訪談問題設計（問聽眾）

**目標：** 驗證使用者真實的行為與動機，聊他們過去的經驗，而不是問他們對未來「假設性」工具的看法。

**探索過去的具體行為（V + P）**
- 回想上一次聽完講座後，心中有深刻想法時，您『後來』都怎麼做？
- 有沒有曾經因為一個想法很深刻，而特別想要記錄下來或分享的經驗？您是如何做的？

**挖掘問題與動機（J）**
- 有沒有過「很想分享，但最後卻沒有做」的情況？為什麼？
- 對您來說，最沒有壓力、最習慣的記錄或溝通方式是什麼？

### 整體策略核心

> 先驗證問題，再設計流程，最後才選擇工具。

這個案例最關鍵的學習：**當客戶來找你「要方案」時，先別急著給方案。先確認他真正面對的問題是什麼。**

## 案例 2：講師回饋整理（完整跑完四步驟）

這是個範本案例，把整套真實需求調查法從假設到決策跑完一輪。

### 步驟 0：核心假設

> 我假設，對於大專任課教師或講師，課後回饋的彙整與重點整理，每週至少一次，平均耗時超過 90 分鐘，目前以 Excel 與訊息截圖處理，重複與無效內容比例高於五成。

假設裡六個元素都齊：
- 目標用戶：大專任課教師或講師
- 痛點描述：課後回饋的彙整與重點整理
- 發生頻率：每週至少一次
- 時間成本：90 分鐘以上
- 現有解決方案：Excel + 訊息截圖
- 問題嚴重程度：重複與無效內容比例高於五成

### 步驟 1：質化探索（5-8 人深度訪談）

**訪談重點**

| 元素 | 問題 |
|---|---|
| V | 「您在處理學生回饋時，最頭痛的是什麼？具體遇到哪些困難？」 |
| J | 「如果這個問題沒解決，對您的教學或工作有什麼影響？會造成什麼損失？」 |
| P | 「您現在都怎麼處理這些回饋？可以詳細描述一下完整流程嗎？用什麼工具？」 |
| D | 「您一週大概有幾堂課？每堂課學生人數多少？教學年資多久？主要教授什麼科目？」 |

### 步驟 2：量化驗證（問卷 100+ 人）

**頻率驗證**
```
您每週需要處理學生回饋的次數
□1次  □2-3次  □4-5次  □6次以上
```

**時間成本**
```
每次整理學生回饋平均花費時間
□30分鐘內  □30-60分鐘  □60-90分鐘  □90分鐘以上
```

**問題嚴重程度**
```
您認為收到的回饋中，重複或無效內容的比例
□不到30%  □30-40%  □40-50%  □50-60%  □超過60%
```

**現有解決方案**
```
目前主要使用的整理工具（可複選）
□Excel  □Word  □Google表單  □截圖整理  □手寫筆記  □其他___
```

**痛點確認**
```
以下哪些是您在處理學生回饋時的主要困擾（可複選）
□內容重複太多  □格式不統一  □難以找到重點
□彙整耗時  □分析困難  □其他___
```

### 步驟 3：訊號分析

收完訪談 + 問卷後，標出每位受訪者的承諾訊號分數。

| 等級 | 範例 |
|---|---|
| 0 | 「這點子真棒！」「應該會有人需要。」 |
| 1 | 「聽起來不錯。」「我可能會用。」 |
| 2 | 「我願意下週安排 30 分鐘看 Demo。」 |
| 3 | 「我可以介紹給系上其他老師。」「我願意在教師會議上分享。」 |
| 4 | 「有教育方案嗎？我現在就可以付費試用。」「學校可以編預算購買。」 |

### 步驟 4：迭代決策

**繼續推進的訊號**
> 多位講師表示願意試用並推薦給同事，甚至主動詢問付費方案和導入時程。

**調整方向的訊號**
> 發現講師真正的痛點是「學生參與度低」或「回饋品質差」，而非回饋整理本身。

**放棄點子的訊號**
> 多數講師表示現有 Excel 處理方式已經足夠，不願意改變既有的工作流程。

## 從這兩個案例可以學到什麼

### 1. 客戶來找你要方案 ≠ 你已經知道問題

基金會案例：對方明確說要「自動化收集回饋系統」，但深挖之後可能發現真正問題是「聽眾根本不想回饋」或「回饋進來沒人處理」。
表面需求 ≠ 真實需求。

### 2. 假設要寫具體，不要寫概念

講師案例：假設不是寫「老師整理回饋很麻煩」，而是寫「每週至少一次、超過 90 分鐘、用 Excel + 截圖、重複內容比例高於五成」。
具體到可以驗證的程度，才是好假設。

### 3. 訪談要分兩種人問

基金會案例同時訪談「決策者（經理）」和「使用者（聽眾）」。
- 決策者告訴你「為什麼想做」「成功長什麼樣」
- 使用者告訴你「實際上會不會用」「現在怎麼做」
兩個都要，缺一個都會出錯。

### 4. 結果可能跟你想的不一樣

跑完流程後，三種結局都有可能：推進、調整、放棄。
重點不是「驗證你的假設是對的」，是「找出真實的需求是什麼」。
願意接受「我的點子是錯的」這個結論，才是這套方法真正的價值。
