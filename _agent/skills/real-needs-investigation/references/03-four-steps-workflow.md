# 四步驟操作流程

從一個模糊的點子，到敢拍板的決策，中間要走完這四步：

```
步驟 0：定義核心假設
    ↓
步驟 1：質化探索（深度訪談 5-8 人）
    ↓
步驟 2：量化驗證（問卷 100+ 人）
    ↓
步驟 3：訊號分析（評估承諾強度 0-4 分）
    ↓
步驟 4：迭代決策（推進／調整／放棄）
```

---

## 步驟 0：定義核心假設

開始之前，先用一句話清晰定義你要驗證的假設。

### 假設要包含的六個元素

1. **目標用戶**：誰會遇到這個問題？
2. **痛點描述**：他們具體在卡什麼？
3. **發生頻率**：多久發生一次？
4. **時間成本**：每次花多久處理？
5. **現有解決方案**：他們現在怎麼處理？
6. **問題嚴重程度**：這個問題佔他們工作 / 生活的比重？

### 假設範本

> 我假設，對於 [目標用戶]，[痛點描述]，[發生頻率]，[時間成本]，目前以 [現有解決方案] 處理，[問題嚴重程度]。

### 案例：講師整理回饋

> 我假設，對於大專任課教師或講師，課後回饋的彙整與重點整理，每週至少一次，平均耗時超過 90 分鐘，目前以 Excel 與訊息截圖處理，重複與無效內容比例高於五成。

### 為什麼要先寫假設？

不寫假設，後面的訪談就會「想到什麼問什麼」，最後拿一堆零散資料。
寫了假設，每個訪談題目都是在驗證假設裡的某個元素，分析時也有對照基準。

---

## 步驟 1：質化探索（深度訪談）

### 規模
- **5-8 位**目標用戶
- 30-60 分鐘半結構化訪談
- 重點收集對方的「原話」和具體場景

### 訪談重點問題（套用 VJPD）

```
V - 驗證問題
「您在處理 OO 時，最頭痛的是什麼？具體遇到哪些困難？」

J - 判斷影響
「如果這個問題沒解決，對您的工作有什麼影響？會造成什麼損失？」

P - 探究行為
「您現在都怎麼處理？可以詳細描述一下完整流程嗎？用什麼工具？」

D - 用戶畫像
「您一週大概有幾堂課？每堂課學生人數多少？教學年資多久？主要教授什麼科目？」
```

### 訪談紀錄要點
- 盡量錄音 + 轉逐字稿（事後分析才能找到關鍵原話）
- 記下每位受訪者的「具體事件」「具體數字」「具體後果」
- 不只記答案，也記他們講話時的情緒（卡住、嘆氣、無奈、興奮）
- 訪談結束前問：「我可以下週再回來訪一次嗎？」 — 這是承諾訊號的隱性測試

### 訪談數量為什麼是 5-8 人

- 少於 5 人 → 樣本太薄，可能撞到極端值
- 5-8 人 → 通常能看出共通模式（80% 的痛點會浮現）
- 多於 8 人 → 邊際效益遞減，先進入步驟 2 量化更划算

---

## 步驟 2：量化驗證（問卷設計）

把訪談發現轉成可量化的問題，對 100+ 目標受眾驗證。

### 問卷範本（以講師整理回饋為例）

**頻率驗證**
```
您每週需要處理學生回饋的次數？
□ 1 次   □ 2-3 次   □ 4-5 次   □ 6 次以上
```

**時間成本**
```
每次整理學生回饋平均花費時間？
□ 30 分鐘內   □ 30-60 分鐘   □ 60-90 分鐘   □ 90 分鐘以上
```

**問題嚴重程度**
```
您認為收到的回饋中，重複或無效內容的比例？
□ 不到 30%   □ 30-40%   □ 40-50%   □ 50-60%   □ 超過 60%
```

**現有解決方案**
```
目前主要使用的整理工具（可複選）：
□ Excel   □ Word   □ Google 表單
□ 截圖整理   □ 手寫筆記   □ 其他___
```

**痛點確認**
```
以下哪些是您處理學生回饋時的主要困擾（可複選）？
□ 內容重複太多   □ 格式不統一   □ 難以找到重點
□ 彙整耗時   □ 分析困難   □ 其他___
```

### 問卷設計要點
- 量化選項要能換算成具體數字（「30-60 分鐘」可以加總算年度時間損失）
- 選項要 mutually exclusive（不要重疊），collectively exhaustive（涵蓋常見情況 + 「其他」）
- 控制題目數量在 8-12 題以內，超過會掉答題率
- 結尾問一題開放題：「您願意進一步聊聊嗎？留下聯絡方式」 — 為下一步篩選候選人

## 步驟 3：訊號分析（承諾強度評分）

分析每位受訪者的承諾訊號強度（詳見 `04-commitment-signals.md`）：

| 等級 | 類型 | 範例 |
|---|---|---|
| 0 分 | 無效訊號 | 「這點子真棒！」 |
| 1 分 | 興趣訊號 | 「我可能會用」 |
| 2 分 | 時間承諾 | 「我願意下週安排 30 分鐘看 Demo」 |
| 3 分 | 名聲承諾 | 「我可以介紹給系上其他老師」 |
| 4 分 | 金錢承諾 | 「有教育方案嗎？學校可以編預算購買」 |

### 分析統計表

| 受訪者 | 日期 | 承諾等級 | 具體訊號 | 備註 |

| A 老師 | 2026-05-01 | 3 | 「願意介紹給系主任」 | 推進 |
| B 老師 | 2026-05-02 | 1 | 「有空試看看」 | 觀察 |
| C 老師 | 2026-05-03 | 4 | 「可以編預算」 | 推進 |
| ... | | | | |

## 步驟 4：迭代決策

根據訊號強度分佈，做出三選一的決策：

### 繼續推進（2-4 分訊號占 60% 以上）
情境：多位受訪者表示願意試用並推薦給同事，甚至主動詢問付費方案和導入時程。

行動：進入 MVP 階段，開始打造最小可行產品 / 服務。

### 調整方向（1-2 分訊號較多，但指向其他痛點）
情境：發現對方真正的痛點不是你假設的那個，而是另一個（例如假設是「整理回饋耗時」，實際是「學生根本不寫回饋」）。

行動：重新定義核心假設，回到步驟 0 再跑一輪。

### 放棄點子（0-1 分訊號占大部分）
情境：多數受訪者表示現有方法已經夠用，不願意改變。

行動：放下這個點子。把資源留給下一個。

## 整套流程的時間配置

| 步驟 | 預估時間 | 產出 |
|---|---|---|
| 步驟 0 | 1-2 小時 | 一句假設陳述 |
| 步驟 1 | 1-2 週 | 5-8 份訪談記錄 |
| 步驟 2 | 1 週 | 問卷上線、收滿樣本 |
| 步驟 3 | 2-3 天 | 訊號統計表 + 結論 |
| 步驟 4 | 半天 | 決策報告 |

整套大約 3-5 週。聽起來慢，但比起你做半年才發現方向錯了，這套流程救你的時間是十倍以上。

## 加速版：簡化流程

不是每個假設都要跑完整四步驟。下列情境可以簡化：

- **小型內部假設**：只跑步驟 0 + 步驟 1（5 人訪談）就夠
- **既有產品的新功能**：可以直接從步驟 2 開始（用現有用戶池跑問卷）
- **完全沒接觸過的市場**：建議完整跑，不要省

判斷原則：
- 假設失敗的代價越大 → 越要跑完整流程
- 試錯成本越低 → 可以越快進入 MVP 階段
