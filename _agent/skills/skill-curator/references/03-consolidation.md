# 整併與拆分細則

## 三種 pattern

### Pattern A：太近的合（Merge）

**判定條件（任一即可考慮）：**
- 兩包功能重疊 ≥ 70%
- 兩包的觸發詞會打架（接模式一結果）
- 兩包都依賴同一份 references／同一個外部工具
- 合併後總篇幅可控（≤ 800 行）

**合併方式：**

1. **以主吞副**：A 是主要使用、B 是衍生小工具 → A 作為合併後的主體，B 的功能變成 A 的一個模式或 references/ 章節
2. **共生新包**：兩包功能對等、合起來能形成更完整能力 → 新建 C 包，A、B 各自貢獻段落
3. **保留兩包，但明確分工**：重疊但分工可以講清楚 → 各自 description 加限定詞，不合併（這算「不合」的 pattern）

**合併後要做：**
- 原 A、B 包 `mv` 到 `_trash/YYYY-MM-DD/`
- 新包通過模式一健檢確認觸發詞乾淨
- 更新所有引用（CLAUDE.md 路由、其他技能包的 wiki-link）

### Pattern B：太雜的拆（Split）

**判定條件（任一即可考慮）：**
- SKILL.md 超過 800 行
- 包含 3 個以上不相關功能
- 不同模式之間幾乎不互相引用
- 觸發詞分散到完全不同領域（例：「寫文章」和「整理財務」同一個包）

**拆分方式：**

1. **核心 + 外掛**：保留核心功能，邊緣功能拆到獨立小包
2. **平行拆**：所有功能對等，拆成 N 個專門包
3. **抽公用層**：多個包都用的部分抽出來變獨立技能包（少用，公開版通常不適合）

**拆分後要做：**
- 原包 `mv` 到 `_trash/YYYY-MM-DD/` 備份
- 新包各自獨立、通過模式一健檢
- 在 CLAUDE.md 與心智圖補新包

### Pattern C：太久沒用的封存（Archive）

**判定條件（同時滿足）：**
- 30 天以上沒被觸發（如果有日記紀錄）
- 對應的工作流已經改變或廢棄
- 但內容可能日後有用，不直接刪

**封存方式：**

- `mv` 到 `_trash/YYYY-MM-DD/` 或專門的 `_archive/` 資料夾
- 保留說明：為什麼封存、日後若要復原該回到哪個技能包

## 不合不拆的情境

以下狀況不要動：

- 兩包功能相似但服務不同使用者群 → 保留兩個
- 包很大但結構清楚、references/ 完整 → 不算「太雜」，不拆
- 30 天沒用但屬於季節性工具（例：年度報告、季度復盤） → 不封存
- 使用者最近正在迭代中的包 → 不動

## 盤點表格式

```
📋 技能包盤點
總數：N

| 名稱 | SKILL 行數 | references 檔數 | 最近修改 | 最近觸發 | 標記 |
|------|-----------|----------------|---------|---------|------|
| A    | 320       | 3              | 5/10    | 5/12    | 健康 |
| B    | 850       | 1              | 4/02    | 5/01    | 🟡太雜 |
| C    | 180       | 0              | 2/15    | 60+ 天 | 📦封存候選 |
| D    | 450       | 4              | 5/08    | 5/11    | 🤝跟 E 重疊 80% |
| E    | 380       | 2              | 5/09    | 5/12    | 🤝跟 D 重疊 80% |
```

## 整併計畫輸出格式

```
📋 整併與拆分計畫
盤點時間：YYYY-MM-DD

🤝 建議合併（M 組）
- D 包 + E 包 → 新包 F
  理由：功能重疊 80%、觸發詞「寫脆」「寫推文」打架
  合併方式：以 D 為主吞 E，E 特殊功能放 F/references/extra.md

✂️ 建議拆分（K 個）
- B 包（850 行）拆三：
  - B-core（核心功能，~400 行）
  - B-mode-X（功能 X，~250 行）
  - B-mode-Y（功能 Y，~200 行）

📦 建議封存（L 個）
- C 包：60+ 天未觸發、舊工作流
  處理：mv _trash/YYYY-MM-DD/
  備份說明：未來若回到 XXX 工作流可復原

—— 下一步 ——
使用者勾選 → 轉技能包大師執行
```
