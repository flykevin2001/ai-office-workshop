# 更新檢查細則

## 核心理念再強調

技能包不能無限膨脹。模式四不是「掃出新規則就寫進去」，是「先問值不值得記，再問該擠掉誰」。

寧可少寫，不亂塞。

## 範圍與成本對照

| 範圍 | 預估時間 | 適合時機 |
|------|---------|---------|
| 1. 🎯 特定某篇技能包 | 短 | 你覺得某個包跟實際做的不一樣 |
| 2. 📓 最近工作日記 | 中 | 定期週檢、想看最近有沒有新規則冒出 |
| 3. 📁 最近常用專案 | 較長 | 大專案告一段落、要把學到的東西沉澱 |
| 4. 🌐 整個知識庫 | 長 | 建議每週用量還剩很多、感覺用不完時再做 |

啟動前一定要先問使用者選哪個範圍，不要擅自決定。

## Gate 1：值不值得記

從最近材料抓出候選規則。每條問三題：

### 題 1：重複出現過 ≥2 次？

- 在不同日記、不同對話、不同檔案被提到過至少兩次
- 一次提到不算規則，是事件
- 證據必須附檔名 + 行號 + 引用原句

### 題 2：寫進技能包後，未來會被 AI 用到？

- 是判斷依據（AI 在做某件事時會用到）？→ yes
- 是純記錄、純歷史？→ no
- 是私人偏好但跨情境通用？→ yes
- 是一次性偏好（這次想這樣下次不一定）？→ no

### 題 3：是規則／決策，不是當下狀態？

- 「以後寫脆文都加 X」→ 規則 → yes
- 「這週狀態不好」→ 狀態 → no
- 「Y 工具改名了，現在叫 Z」→ 事實更新 → yes
- 「現在正在做 W 專案」→ 當下進行 → no

**三題都 yes 才進 Gate 2，否則丟。**

## Gate 2：要擠掉誰

每個技能包軟上限：
- SKILL.md ≤ 500 行
- references/ 單檔 ≤ 300 行
- 整個技能包 references/ 總檔數 ≤ 8 個

新規則要寫進去，必須回答以下任一：

### 選項 A：找一段過時的刪

- 哪一段最近 30 天沒被引用、跟新規則矛盾、或描述的工作流已經改了？
- 直接刪掉，騰出空間

### 選項 B：找一段濃縮

- 某段太冗長、可以壓到一半篇幅？
- 濃縮後騰出空間給新規則

### 選項 C：找正確分層

- 這條規則是「決策邏輯」還是「操作細節」？
- 操作細節 → 下放到 references/
- 跨技能包通用 → 上提到 CLAUDE.md
- 不在現有技能包範疇 → 新建技能包（轉技能包大師）

### 沒人能擠掉怎麼辦

該規則暫不寫入，進「候選池」（一份獨立的 markdown 檔，每個技能包對應一份）。

下次模式四再評。

- 連續通過 3 次 → 強制找空間寫入（可能要拆分原包）
- 連續 3 次都沒被寫入 → 永久丟棄

## 掃描方法

### 範圍 1：特定某篇技能包

```bash
# 找出引用該技能包的日記與對話
grep -rln "技能包名稱" 工作日記/ 對話紀錄/
```

讀對應的檔案，比對 SKILL.md。

### 範圍 2：最近工作日記

```bash
find 工作日記/ -name "*.md" -mtime -14
```

讀近 14 天日記，找重複出現的規則／偏好／流程描述。

### 範圍 3：最近常用專案

```bash
# 找近期高頻修改的資料夾
find . -type f -name "*.md" -mtime -30 | awk -F/ '{print $1"/"$2}' | sort | uniq -c | sort -rn | head -10
```

針對 top 5 資料夾，掃內容。

### 範圍 4：整個知識庫

不建議用 find 全掃，會吃太多 token。建議：

- 先用 git log 或檔案修改時間排序，挑近 60 天有變動的
- 分批掃，一批 10-20 個檔案
- 跑完一批就出中間報告，使用者確認後再跑下一批

## 偵查報告格式

```
📋 更新檢查報告
範圍：[使用者選的範圍]
時間：YYYY-MM-DD ~ YYYY-MM-DD
掃描檔案數：XX 篇
候選規則：12 條 → 通過 Gate 1：5 條 → 通過 Gate 2：3 條

✅ 建議寫入（3 條）

1. [規則內容]
   證據：
   - 日記 2026-05-08 行 23：「⋯⋯」
   - 對話 2026-05-10：「⋯⋯」
   - 檔案 X 行 45：「⋯⋯」
   寫入：spring-editor SKILL.md
   代價：刪除 SKILL.md 第 X 段（已過時，最近一次提到 30 天前）

2. [規則內容]
   ⋯⋯

3. [規則內容]
   ⋯⋯

⏸️ 候選池（2 條，本次不寫）

- [規則內容]
  原因：找不到能擠掉的段落，留待下次
  證據：⋯⋯

🗑️ 直接丟（7 條）

- [簡短列表]
  原因：重複不足 / 不會再用 / 屬於當下狀態

—— 下一步 ——
使用者勾選哪幾條要寫入 → 轉技能包大師執行
```

## 守則

- 寧可少寫，不亂塞
- 每條新規則都要回答「擠掉誰」
- 候選池每月重檢一次
- 沒有證據 → 寧可不報，不亂編
- 範圍由使用者決定，不擅自擴大
- 跑完一個範圍記一份工作日記（範本見 diary-template.md）
