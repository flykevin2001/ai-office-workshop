# 技能包大總管 v1.0

公開版。把整個工作流裡的技能包當成一個有機體來治理。

技能包大師（skill-master）管「怎麼做好一個包」。
技能包大總管（這個包）管「整個工作流裡這些包該怎麼共存」。

兩個分開。前者是作者工具，後者是池子治理工具。

---

## 為什麼需要這個技能包

當你開始用 AI agent 工作流，技能包會自然累積。三個來源同時冒：

1. **自己寫的**（用技能包大師做的）
2. **網路抓的**（公開倉庫、別人分享的）
3. **AI 順手生的**（對話中產生的小技能）

久了會出三種問題：

- **觸發詞打架**：兩個技能包都用「整理」「歸檔」當觸發詞，AI 不知道叫哪個
- **重複堆積**：A 包跟 B 包做的事 80% 重疊，但都還在
- **規格漂移**：實際工作流變了，但技能包 SKILL.md 還是舊的，AI 用了老地圖

這個技能包就是處理這三件事——加上一個閉環機制：模式四會掃最近工作日記，主動偵測技能包該不該更新。

---

## 四種工作模式

| 模式 | 觸發條件 | 做什麼 |
|------|---------|------|
| 模式一：觸發詞健檢 | 「健檢觸發詞」「觸發詞會不會打架」 | 掃所有技能包 description，列衝突清單 + 收斂建議 |
| 模式二：Skill 快篩 | 「Skill 快篩」「這個技能包要不要收」「網路抓到一個 X」 | 跟現有工作流比對 → 收 / 調整後收 / 不收 |
| 模式三：整併與拆分 | 「整理我的知識庫技能包」「這兩個是不是該合併」 | 找太近的（合）、太雜的（拆）、太久沒用的（封存） |
| 模式四：更新檢查 | 「檢查技能包要不要更新」「最近工作有沒有沒寫進去的」 | 掃最近材料 → 兩道 gate 過濾 → 列建議寫入 / 候選 / 丟棄 |

完整生命週期：篩 → 健檢 → 用 → 偵測漂移 → 整併或更新 → 再健檢

---

## 共通守則（四模式都適用）

### 一、只列計畫，不擅自動手

任何改動（合併、拆分、刪除、改名、寫入新規則）都先列計畫給使用者確認。確認後才動。

### 二、刪除前先搬到 `_trash/`

不直接 `rm`。要刪的技能包先 `mv` 到 `_trash/YYYY-MM-DD/`，使用者自行清理。

### 三、每次整理留工作日記

格式參考 `references/diary-template.md`。下次盤點時可以對照看軌跡。

### 四、依賴個人設定檔

模式二（Skill 快篩）特別依賴個人設定檔（user-profile）。沒有 profile 就無從判斷一個技能包對使用者適不適合。詳見模式二段落。

### 五、不取代技能包大師

這個包只做篩選、整併、健檢、更新偵測。實際創建、修改 SKILL.md 的工作是技能包大師的事。本包列出待動清單後，轉給技能包大師執行。

## 模式一：觸發詞健檢

### 觸發條件

使用者說：「健檢觸發詞」「觸發詞會不會打架」「我的技能包觸發詞會不會搶」「最近 AI 老是叫錯技能包」

### 核心原則

**一個技能包配 1-2 個清楚的主觸發詞。**

過多觸發詞會降低 AI 判斷效率，也增加跟其他技能包衝突的機會。技能包大師建議 description 寫得稍微積極一點，但「積極」指描述具體、不是觸發詞數量多。

### 執行步驟

1. **掃描所有技能包的 description**
   - 路徑：`_agent/skills/*/SKILL.md`
   - 抓出 frontmatter 裡 description 欄位
   - 抓出明顯的觸發詞（「當使用者說...」「觸發詞...」等段落）

2. **建立觸發詞清單**
   - 一個技能包對應一個觸發詞集合
   - 用表格呈現：技能包名稱 / 觸發詞 / 數量

3. **找衝突**
   - 同一個詞出現在兩個以上技能包 → 衝突
   - 近義詞（例：「整理」vs「歸檔」、「寫」vs「製作」）→ 疑似衝突
   - 一個技能包超過 5 個觸發詞 → 太多，建議收斂

4. **產出健檢報告**
   ```
   📋 觸發詞健檢報告
   掃描時間：YYYY-MM-DD HH:MM
   技能包總數：N
   
   🔴 直接衝突（M 組）
   - 「整理」同時用於 A、B、C 三個技能包
     建議：A 改用「歸檔」、B 改用「分類」、C 保留「整理」
   
   🟡 疑似衝突（K 組）
   - 「寫」vs「製作」近義
     涉及：D（寫文章）、E（製作圖文卡）
     建議：D 加限定「寫脆文」「寫方格子」，E 保留
   
   🟢 觸發詞過多（L 個）
   - F 包有 8 個觸發詞，建議收斂至 2 個主觸發 + 3 個替代
   ```

5. **使用者勾選哪些要動 → 轉技能包大師更新 description**

詳細衝突判定規則：`references/01-trigger-health-check.md`

## 模式二：Skill 快篩

### 觸發條件

使用者說：「Skill 快篩」「這個技能包要不要收」「網路抓到一個 X」「GitHub 上看到一個技能包」

### 前置依賴：個人設定檔

**沒有個人設定檔，這個模式無法正確運作。**

為什麼依賴 profile：
1. **判斷標準不能憑空**：「這包適不適合」要先知道使用者在做什麼、用哪些工具、看重什麼
2. **避免一刀切**：同一個技能包對 A 是寶、對 B 是垃圾
3. **跟現有工作流對齊**：profile 裡列現有技能包與職責，快篩才能判斷重複／互補／衝突
4. **責任分層**：profile 管「我是誰」，curator 管「這包對我這個身份適不適合」

個人設定檔技能包（user-profile）由使用者自己建。本包只負責讀取與使用。

### 執行步驟

**Step 1：確認個人設定檔存在**

- 沒有 profile → 提示使用者：
  ```
  Skill 快篩需要先建個人設定檔，知道你是誰、在做什麼，才能判斷適不適合。
  
  選一：
  A) 建立完整個人設定檔（推薦，未來其他技能包也會用到）
  B) 跑「快速 mini-profile」（只服務這次快篩，跑完丟掉）
  ```

- B 選項的三題：
  1. 你主要想做什麼？
  2. 想要達到什麼效果、目標？
  3. 最怕踩什麼坑？

- 有 profile → 讀取，進入 Step 2

**Step 2：載入待篩選的新技能包**

使用者貼：
- GitHub link
- 或檔案路徑
- 或口頭描述

至少要拿到 SKILL.md 的 name + description + 模式列表。

**Step 3：三維度比對**

維度 1：跟 profile 工作流的匹配度
- 這包做的事在使用者的主要目標路徑上嗎？
- 還是繞遠路？

維度 2：跟現有技能包的重疊度
- 跟哪些現有技能包功能重疊？
- 重疊比例多少？

維度 3：跟使用者判斷偏好的契合度
- 使用者最怕的坑這包會不會踩？
- 風格、語氣、自動化程度合不合？

**Step 4：三選一決定**

```
✅ 直接收 — 跟工作流匹配、跟現有無重疊、契合使用者偏好
   下一步：放 _agent/skills/，更新 CLAUDE.md 路由表

🟡 調整後收 — 大方向對，但有 X 個地方要改
   下一步：列出該改的點，轉技能包大師執行修改

❌ 不收 — 跟現有 Y 重疊太高 / 不在工作流目標路徑 / 會踩到 Z 坑
   下一步：告訴使用者拒絕理由，建議使用既有的哪一個替代
```

詳細評估清單：`references/02-skill-quick-screen.md`

## 模式三：整併與拆分

### 觸發條件

使用者說：「整理我的知識庫技能包」「這兩個是不是該合併」「技能包好像太多了」「有些好久沒用了」

### 三種常見 pattern

**Pattern A：太近的合（merge）**
- 兩個技能包功能重疊 ≥70%
- 觸發詞會打架
- 合併後不會變得太雜（總篇幅可控）

**Pattern B：太雜的拆（split）**
- 一個技能包包了 3 個以上不相關功能
- SKILL.md 超過 800 行
- 不同模式之間幾乎不互相引用

**Pattern C：太久沒用的封存（archive）**
- 30 天以上沒被觸發
- 對應的工作流已經改變或廢棄
- 但內容可能日後有用，不直接刪

### 執行步驟

1. **盤點所有技能包**
   - 列出：名稱、最後修改時間、最近一次出現在工作日記的時間（如果有日記的話）
   - 算出每包的 SKILL.md 行數、references/ 檔案數
   - 用表格呈現

2. **套用三個 pattern 篩出候選**

3. **產出整理計畫**
   ```
   📋 技能包整併與拆分計畫
   
   🤝 建議合併（M 組）
   - A 包 + B 包 → 新包 C
     理由：功能重疊 80%、觸發詞衝突
     合併方式：A 為主，B 的特殊功能放 C/references/
   
   ✂️ 建議拆分（K 個）
   - D 包過大（1200 行），包含三件事
     拆法：D-core（保留主要功能）、D-extra-1、D-extra-2
   
   📦 建議封存（L 個）
   - E 包 60 天未觸發
     處理：mv 到 _trash/YYYY-MM-DD/，可日後復原
   ```

4. **使用者勾選哪些要動 → 轉技能包大師執行**

詳細 pattern 判定：`references/03-consolidation.md`

## 模式四：更新檢查

### 觸發條件

使用者說：「檢查技能包要不要更新」「最近工作有沒有沒寫進去的」「我覺得 SKILL.md 跟我實際做的不太一樣」

### 核心理念：有界記憶

**技能包不能無限膨脹。** 每次更新都是取捨：新東西要進，舊東西要擠出去，或乾脆不寫。

不是「掃出新規則就寫進去」。是「先問值不值得記，再問該擠掉誰」。

### 啟動前必先警告

```
⚠️ 這是大工程，先選範圍：

1. 🎯 特定某篇技能包      — 掃單包 + 對應日記
2. 📓 最近工作日記        — 預設近 14 天
3. 📁 最近常用專案        — 列出近期高頻資料夾
4. 🌐 整個知識庫          — 建議每週用量還剩很多、感覺用不完時再做

請選 1/2/3/4，或自己指定範圍（例如「掃 5/1 到今天的日記」「只看 spring-editor」）
```

### 兩道 Gate

**Gate 1：值不值得記？**

從最近材料找出候選規則 → 每條問三題：

1. 重複出現過 ≥2 次？（一次不算規則，是事件）
2. 寫進技能包後，未來會被 AI 用到？（不會用到的不寫）
3. 是規則／決策，不是當下狀態？（狀態類的擠掉就過時了）

三題都 yes 才進 Gate 2，否則丟。

**Gate 2：要擠掉誰？**

每個技能包預設有軟上限：
- SKILL.md ≤ 500 行
- references/ 單檔 ≤ 300 行

新規則要進去，必須明確指出：

- 哪一段已經過時可以刪？
- 哪一段可以濃縮？
- 還是這條規則該下放到 references/、或上提到 CLAUDE.md？

**沒人能擠掉 → 該規則暫不寫入**，進「候選池」，下次再評。

### 偵查報告格式

```
📋 更新檢查報告
範圍：[使用者選的範圍]
時間：YYYY-MM-DD ~ YYYY-MM-DD
掃描檔案數：XX 篇
候選規則：12 條 → 通過 Gate 1：5 條 → 通過 Gate 2：3 條

✅ 建議寫入（3 條）
- [規則內容]
  證據：日記 2026-05-08、對話 2026-05-10、檔案 X
  寫入：spring-editor SKILL.md
  代價：刪除 SKILL.md 第 X 段（已過時，最近一次提到 30 天前）

⏸️ 候選池（2 條，本次不寫）
- [規則內容]
  原因：找不到能擠掉的段落，留待下次

🗑️ 直接丟（7 條）
- [簡短列表]
  原因：重複不足 / 不會再用 / 屬於當下狀態
```

### 守則

- 寧可少寫，不亂塞
- 每條新規則都要回答「擠掉誰」
- 候選池每月重檢一次，連續三次沒被寫入 → 永久丟棄
- 沒有證據 → 寧可不報，不亂編
- 範圍由使用者決定，不擅自擴大

詳細掃描方法：`references/04-update-check.md`

## 與其他技能包的關係

| 技能包 | 關係 |
|------|------|
| skill-master（技能包大師） | 互補。本包列出待動清單，由 skill-master 執行修改、創建、評估 |
| user-profile（個人設定檔） | 強依賴。模式二必須讀 profile 才能判斷篩選 |
| 其他所有技能包 | 治理對象。本包不修改任何技能包，只列建議 |

---

## 工作日記

每次跑完任一模式，都建議留一份工作日記，預設位置：`池子工作日記/YYYY-MM-DD-HHMM 技能包大總管：模式X 報告.md`

公開版不強制位置，使用者可指定。範本見 `references/diary-template.md`。
