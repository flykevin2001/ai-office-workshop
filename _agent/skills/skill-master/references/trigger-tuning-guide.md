# 模式五：觸發調校 操作指南

這是技能包大師 v3.0 模式五的完整操作指南。底層依賴 Anthropic 官方 skill-creator 的觸發優化引擎。

---

## 概念說明

### 觸發機制怎麼運作的？

AI 看到使用者的提示詞後，會拿它跟所有已安裝技能包的 name + description 比對，決定要不要觸發某個技能包。

關鍵：AI 只看 YAML frontmatter 裡的 name 和 description，不看 SKILL.md 的內容。所以 description 寫得好不好，直接決定技能包會不會被正確觸發。

### 目前的觸發傾向

AI 傾向「少觸發」而不是「多觸發」。原因是 AI 如果判斷自己就能處理（不需要額外指引），它就不會去讀技能包。

這意味著：
- description 寫得太保守，技能包就不會被觸發
- 簡單的任務即使描述完全匹配也可能不觸發
- 稍微積極一點的 description 反而更好

### 為什麼要調校？

當你的技能包集合越來越大（超過 10 個），觸發問題就會浮現：
- 該觸發的沒觸發
- 不該觸發的觸發了（搶到別的技能包的任務）
- 相似功能的技能包互相打架

## 操作步驟

### 第一步：設計觸發測試查詢

準備 20 個測試查詢：8-10 個應觸發（should_trigger: true）+ 8-10 個不應觸發（should_trigger: false）。

設計原則：

應觸發的查詢：
- 用不同的語氣和用詞描述同一個需求
- 包含正式和隨意的表達方式
- 包含不直接點名技能包但明確需要它的情境
- 包含跟其他技能包會競爭但應該由這個技能包處理的情境

不應觸發的查詢：
- 最有價值的是「近似但不同」的查詢，跟技能包的關鍵字有重疊但實際上不需要它
- 包含相鄰領域的任務
- 包含看起來像但其實用另一個工具更適合的情境

怎樣算好的測試查詢：
- 夠具體，有背景脈絡（不是「幫我做 X」這種抽象的）
- 像真人會打的字（有口語化、有錯字也沒關係）
- 長短不一

怎樣算差的測試查詢：
- 太抽象：「格式化資料」「提取文字」
- 太明顯：用「不應觸發」來放一個完全無關的查詢（例如用「寫費波那契函數」來測 PDF 技能包）

存檔格式：

```json
[
  {"query": "ok 我老闆剛寄了一份 excel 過來，裡面有上個月的銷售數據，她要我加一欄毛利率，收入在 C 欄成本在 D 欄", "should_trigger": true},
  {"query": "可以幫我把這個 csv 轉成 json 格式嗎，我要用在 API 裡面", "should_trigger": false}
]
```

### 第二步：跟使用者確認

用官方提供的 HTML 模板讓使用者檢視和調整查詢。

做法：
1. 讀取 skill-creator/assets/eval_review.html
2. 替換三個 placeholder：
   - `__EVAL_DATA_PLACEHOLDER__` → JSON 陣列
   - `__SKILL_NAME_PLACEHOLDER__` → 技能包名稱
   - `__SKILL_DESCRIPTION_PLACEHOLDER__` → 當前 description
3. 寫到暫存檔，給使用者打開
4. 使用者調整完點「Export Eval Set」匯出

這一步不能跳過。如果測試查詢設計得不好，後面的優化就沒有意義。

### 第三步：跑優化迴圈

#### 自動化（Claude Code / Cowork）

用官方腳本：

```bash
python -m scripts.run_loop \
  --eval-set <path-to-trigger-eval.json> \
  --skill-path <path-to-skill> \
  --model <model-id> \
  --max-iterations 5 \
  --verbose
```

model 要用跟當前對話相同的模型 ID，這樣觸發測試才能反映真實使用情境。

腳本會自動：
1. 把查詢集拆成 60% 訓練集 + 40% 測試集
2. 用訓練集評估當前 description（每個查詢跑 3 次取穩定結果）
3. 用 Claude 的延伸思考能力分析失敗案例，提出改進
4. 用訓練集和測試集分別評估新 description
5. 最多跑 5 輪，選測試集分數最高的版本（避免過擬合）

跑的時候可以定期看 log，跟使用者更新進度。

#### 手動（Claude.ai 或其他不支援腳本的環境）

1. 自己分析 description 的弱點
2. 用技能包的核心功能和觸發情境重寫 description
3. 沒辦法量化測試，但可以用原則來改進：
   - 覆蓋率：使用者可能用的詞都有提到嗎？
   - 區分度：跟其他技能包的差異有說清楚嗎？
   - 積極性：有沒有主動說「即使使用者沒有明確提到 X，只要情境涉及 Y 就該用」？

### 第四步：套用結果

把最佳 description 更新到 YAML frontmatter。

展示前後對比：
- 舊版 description
- 新版 description
- 訓練集分數變化
- 測試集分數變化

## description 撰寫原則

不管用自動化還是手動，好的 description 都遵循這些原則：

### 1. 說清楚做什麼

不要只寫「處理 PDF」。要寫「提取 PDF 文字和表格、合併 PDF、拆分 PDF、填寫 PDF 表單、OCR 掃描」。越具體，AI 越容易判斷。

### 2. 列出觸發條件

明確寫出使用者會用什麼詞。例如：「當使用者說『整理逐字稿』『幫我整理』『做成教學手冊』時觸發。」

### 3. 寫積極一點

因為 AI 傾向少觸發，所以你可以主動擴大觸發範圍。例如：「即使使用者沒有明確說『技能包』，只要他在討論 AI 提示詞的設計方法，就應該觸發這個技能包。」

### 4. 主動區分

如果有容易混淆的技能包，在 description 裡就要區分。例如：「注意：會議記錄請使用會議紀錄技能包，不要用逐字稿整理技能包。」

### 5. 控制長度

description 不宜太長，100-200 字左右。太長反而讓 AI 難以快速判斷。

## 常見觸發問題與解法

### 問題 1：技能包完全沒被觸發

可能原因：
- description 寫得太保守
- 使用者的用詞跟 description 的關鍵字差很遠

解法：
- 加入更多同義詞和口語化表達
- 加入「即使...也該觸發」的句子

### 問題 2：被其他技能包搶走

可能原因：
- 兩個技能包的 description 太相似
- 沒有主動區分

解法：
- 在兩個技能包的 description 裡都加上區分說明
- 強調各自的獨特價值

### 問題 3：不該觸發的時候觸發了

可能原因：
- description 寫得太廣泛
- 缺少「不適用情境」的說明

解法：
- 加入「不要在 XXX 情境下使用」的提示
- 縮小觸發範圍到核心使用情境

## 環境差異對照

| 步驟 | Claude Code | Cowork | Claude.ai |
|------|------------|--------|-----------|
| 設計查詢 | 手動 + HTML 編輯器 | 手動 + HTML 輸出 | 手動 |
| 使用者確認 | HTML 互動 | 靜態 HTML | 對話中確認 |
| 優化迴圈 | run_loop.py 自動化 | run_loop.py 自動化 | 手動改寫 |
| 評估結果 | 自動化報告 | 自動化報告 | 無量化數據 |

---

## 什麼時候需要重新調校？

- 新增了其他技能包（可能搶觸發）
- 技能包改版後功能範圍變了
- 使用者回報「叫不到」或「叫錯了」
- 模型更新後觸發行為改變
