# 為什麼技能包大師需要規格驅動？

## 核心問題：它是「開發工具的開發工具」

```
【第三層】最終使用者
    ↓ 使用
【第二層】技能包（例如：規格驅動開發教練）
    ↓ 由誰產出？
【第一層】技能包大師 ← 這是源頭
    ↓ 由誰產出？
【第零層】你 + Claude
```

技能包大師不是普通工具，它是「產出其他工具」的工具。

這個特殊定位，決定了它必須用規格驅動的方式開發與維護。

---

## 原因一：誤差會放大

### 場景

你用技能包大師產出「規格驅動開發教練」技能包
→ 使用者用「規格驅動開發教練」學習寫規格
→ 使用者用學到的方法開發系統
→ 系統交給使用者使用

### 問題

**如果技能包大師在第一層就偏了 10%：**
- 產出的技能包可能偏 20%
- 使用者學到的方法可能偏 40%
- 最終系統可能偏 80%

**比喻：**

技能包大師像是「模具的模具」：
- 用它做出技能包（第一層模具）
- 技能包產出工作成果（第二層產品）

如果「模具的模具」有瑕疵，所有下游產品都會有瑕疵。

### 解法：規格驅動

**規格是精準度的保證**

- 規格定義清楚，產出才不會偏
- 規格可以被檢視、被驗證
- 發現問題改規格，重新產出即可

**就像機械製造：**
- 模具要有設計圖（規格）
- 按圖施工，精準度有保證
- 圖錯了改圖，不是憑感覺改模具

---

## 原因二：自我繁殖需要穩定迭代

### 哲學問題

技能包大師本身也是一個技能包，這產生一個矛盾：
- 技能包大師用來產出技能包
- 但技能包大師本身也需要被產出
- **那誰來產出技能包大師？**

### 傳統做法的問題

**手動維護：**
```
你手動寫技能包大師 v1.0
    ↓
用 v1.0 產出其他技能包
    ↓
要升級時，手動改 v1.0 → v1.1
    ↓
越改越複雜，越來越難維護
```

**問題：**
1. 手動改容易不一致
2. 改 A 功能時不小心動到 B 功能
3. 版本越多，越難記得改了什麼
4. 無法確保每次產出的品質穩定

### 規格驅動的解法

**自我繁殖：**
```
寫一份「技能包大師的規格」（SPEC.md）
    ↓
用規格產出技能包大師 v3.0
    ↓
要升級時，改 SPEC.md → 重新產出 v3.1
    ↓
甚至可以用 v3.0 產出自己的 v3.1
```

**好處：**
1. 規格是唯一真相來源
2. 改規格 → 重新產出，邏輯一致
3. 版本管理清楚（規格有版本號）
4. **技能包大師可以產出自己的下一版**

**這就像生物的 DNA：**
- DNA 是生物的「規格」
- 生物用 DNA 複製自己
- DNA 突變（改規格）→ 產出新版本
- 穩定的遺傳機制

---

## 原因三：標準制定者的責任

### 技能包大師定義了什麼？

它不只是工具，更是**標準制定者**：

1. **什麼是「好的技能包」**
   - 結構應該長怎樣
   - 必須包含哪些部分
   - 格式規範是什麼

2. **技能包設計方法論**
   - 怎麼定義邊界
   - 怎麼設計觸發條件
   - 怎麼規劃工作流程

3. **品質標準**
   - 怎麼檢查完整性
   - 怎麼驗證可用性
   - 怎麼確保一致性

### 如果沒有規格會怎樣？

**標準不透明：**
- 其他人不知道「好技能包」的標準是什麼
- 無法學習、無法複製
- 只能靠「感覺」判斷好壞

**標準不穩定：**
- 這次產出的技能包是這樣
- 下次產出的技能包可能不一樣
- 使用者困惑：到底哪個才對？

**無法傳授：**
- 你知道怎麼做好技能包
- 但無法教別人
- 方法論留在你腦中，帶不走

### 有了規格之後

**標準透明：**
- 規格就是「什麼是好技能包」的文件
- 任何人都可以看、可以學
- 標準可以被討論、被改進

**標準穩定：**
- 按規格產出，品質穩定
- 每次產出都一致
- 使用者知道可以期待什麼

**可以傳授：**
- 規格就是教材
- 其他人可以學你的方法
- **方法論可以標準化、可以規模化**

**比喻：**

技能包大師像是「料理學校」：
- 規格 = 教學大綱
- 產出技能包 = 培養廚師
- 有大綱才能確保每個廚師都學到標準方法

---

## 原因四：複雜度管理

### 技能包大師越來越複雜

**現在有：**
- 五種模式（引導探索型、專業整理型、規格驅動型、評估與優化、觸發調校）
- 技能包養成法（生命週期管理）
- 判斷觸發條件
- 版本號管理
- 多檔案產出（SKILL.md + references）
- 格式一致性檢查
- 中文化處理規則
- 整合官方 eval / benchmark / trigger tuning 引擎

**未來可能還要：**
- 支援多人協作
- 跨平台部署（Claude Skills → ChatGPT GPTs）
- 技能包市集與分享機制

### 如果沒有規格

**加新功能時：**
- 不知道會不會破壞舊功能
- 要翻遍所有程式碼確認
- 改了 A 發現 B 壞了
- 越改越亂

**邏輯越來越複雜：**
- 功能之間互相影響
- 連你自己都搞不清楚
- 維護成本越來越高
- 最後只能重寫

### 有了規格 + references/ 模組化

**標準架構：**
```
skill-master/
├── SPEC.md               ← 單一設計圖（設計意圖 + 邊界 + 驗收基準）
├── SKILL.md              ← 從規格產出的主檔案
└── references/
    ├── why-spec-driven.md
    ├── skill-spec-template.md
    ├── skill-writing-guide.md
    ├── eval-guide.md
    └── trigger-tuning-guide.md
```

**加新功能時：**
- 先更新 SPEC.md 的設計意圖和邊界
- 對應的操作細節放到 references/ 裡
- SKILL.md 只更新決策邏輯和入口指引
- **SPEC.md 控全局，references/ 管細節，互不干擾**

**維護容易：**
- SPEC.md 是唯一的設計真相來源
- references/ 各自獨立，改某個操作指南不影響其他
- 改完重新產出即可

**比喻：**

技能包大師像是「作業系統」：
- 核心邏輯 = kernel
- 各模式 = 模組
- 規格 = 系統架構文件
- 有文件才能確保系統穩定

---

## 四個原因的關聯

```
原因一：誤差會放大
    ↓ 所以需要
精準的規格
    ↓ 但規格要能
穩定迭代（原因二）
    ↓ 而且要能
傳授給其他人（原因三）
    ↓ 同時要能
控制複雜度（原因四）
```

**結論：**

這四個原因互相支撐，缺一不可。

技能包大師作為「開發工具的開發工具」，必須用規格驅動才能：
1. 確保精準（源頭不能偏）
2. 穩定迭代（自我繁殖）
3. 標準透明（可傳授）
4. 複雜可控（可維護）

---

## 實際影響

### 對你（創建者）的影響

**短期：**
- 第一次寫規格多花 30-60 分鐘
- 要學會「規格驅動思維」

**長期：**
- 每次更新省 2-3 小時（不用來回改）
- 版本管理清楚（知道改了什麼）
- 品質穩定（不會越改越亂）
- **可以用技能包大師產出自己的下一版**

### 對使用者的影響

**品質穩定：**
- 用技能包大師產出的技能包品質一致
- 不會這次很好、下次很爛

**可預期：**
- 知道技能包會長什麼樣
- 知道該包含什麼部分
- 學習曲線降低

**可信賴：**
- 方法論透明（有規格可看）
- 標準清楚（不是黑盒子）
- 可以學習、可以複製

### 對生態系的影響

**方法論可傳播：**
- 其他人可以學你的方法
- 可以一起改進規格
- **技能包設計變成可教、可學的知識**

**標準化：**
- 大家用同樣的規格模板
- 技能包品質提升
- 生態系更健康

**規模化：**
- 你的方法可以被複製
- 不需要你親自產出每個技能包
- **培養更多「技能包設計師」**

## 與 Spec-Kit 的對比

### Spec-Kit 為什麼需要規格驅動？

Spec-Kit 是「產出程式開發工具」的工具：
- 它產出 API、測試、文件
- 它定義了「程式開發的標準流程」
- 它是程式設計的源頭工具

**跟技能包大師一樣的邏輯：**
- 都是「開發工具的開發工具」
- 都需要精準（誤差會放大）
- 都需要穩定迭代（自我繁殖）
- 都需要標準透明（可傳授）

### 技能包大師 vs Spec-Kit

| 比較項目 | Spec-Kit | 技能包大師 |
|---------|---------|-----------|
| 產出對象 | 程式、API、測試 | 技能包 |
| 規格格式 | Markdown + YAML | Markdown |
| 使用對象 | 工程師、技術人員 | 知識工作者 |
| 核心價值 | 程式開發自動化 | 知識產品化 |

**共通點：**
- 都用規格驅動
- 都是開發工具的開發工具
- 都需要高精準度
- 都支援自我繁殖

---

## 結語

技能包大師需要規格驅動，不是為了趕時髦，而是因為：

1. **它是源頭工具**，誤差會放大，必須最精準
2. **它要自我繁殖**，需要穩定的迭代機制
3. **它是標準制定者**，邏輯必須透明可傳授
4. **它會越來越複雜**，需要模組化管理

**這不是錦上添花，而是必要條件。**

當你用規格驅動的技能包大師產出其他技能包時，你會發現：
- 產出速度變快
- 品質更穩定
- 維護更容易
- **整個知識產品化的流程被標準化了**

這就是「開發工具的開發工具」的特殊性與責任。
