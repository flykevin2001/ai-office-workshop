# 技能包寫作指引（延伸說明）

SKILL.md 主檔案已涵蓋寫作指引的核心原則。這份文件提供更多細節和範例。

---

## Progressive Disclosure 延伸

### 為什麼要控制 SKILL.md 長度

技能包被觸發時，SKILL.md 整份會載入 context。如果太長：
- 佔用大量 context window，壓縮了使用者的任務內容空間
- 模型需要從大量文字中找到相關段落，增加出錯機率
- 載入時間和 token 成本都增加

500 行是理想值，不是硬限制。超過的時候要有意識地問自己：「這段放在 SKILL.md 裡，是因為每次觸發都需要，還是只有特定情境才需要？」後者應該移到 references/。

### 怎麼決定放 SKILL.md 還是 references/

放 SKILL.md 的：
- 觸發後立刻需要的決策邏輯（什麼情況做什麼）
- 工作流程的主幹（步驟一、二、三的摘要）
- 模式選擇邏輯（引導探索型 vs 規格驅動型等）

放 references/ 的：
- 某個步驟的完整操作細節
- 特定領域的參考資料（例如 AWS 部署指南、特定框架的用法）
- 案例庫
- JSON schema 定義
- 腳本的使用說明

### references/ 內的檔案超過 300 行怎麼辦

在檔案開頭加一個目錄，讓 Claude 可以快速定位要讀的段落：

```markdown
# 評估操作指南

## 目錄
- 設計測試案例 → 第 15 行
- 跑測試 → 第 80 行
- 設計 assertion → 第 140 行
- 評分與彙整 → 第 200 行
- 改進迴圈 → 第 260 行
```

## Explain the Why 延伸

### 什麼時候可以用命令式

有些情況確實需要硬規則：
- 安全相關：「不要在輸出中包含使用者的密碼」→ 這個用 MUST 合理
- 格式強制要求：「JSON 的 key 必須用 snake_case」→ 技術規格不需要解釋
- 已知的模型弱點：「先讀完整份文件再開始分析，因為模型容易只看開頭就下結論」

判斷原則：如果這個規則背後有明確的技術原因或安全原因，而且不需要模型「理解」就能執行，用命令式沒問題。如果這個規則需要模型「判斷」什麼時候適用，就要解釋 why。

### 怎麼測試你的指令夠不夠好

把你寫的技能包指令念給一個聰明但不懂你領域的人聽。如果他能理解「為什麼要這樣做」而不只是「要做什麼」，那你的指令就是好的。

如果他的反應是「好，但為什麼？」→ 你需要補 why。
如果他的反應是「喔，原來是這樣，那我懂了」→ 指令夠好。

## 通用溝通原則

技能包大師可能被各種程度的使用者使用。溝通時：

- 「eval」和「benchmark」可以直接用，但第一次出現時簡短解釋
- 「JSON」和「assertion」等技術詞，先確認使用者的技術背景再決定要不要解釋
- 盡量用具體範例取代抽象定義

## 輸出格式定義的範例模式

### 用模板定義

```markdown
## 報告結構
一律用這個模板：

# [標題]
## 重點摘要
（3 句話以內）
## 關鍵發現
（每項一段，最多 5 項）
## 建議行動
（具體可執行的步驟）
```

### 用 Input/Output 範例定義

```markdown
## commit message 格式

範例 1：
輸入：新增了 JWT 驗證功能
輸出：feat(auth)：實作 JWT 驗證機制

範例 2：
輸入：修正了使用者登入時的空指標錯誤
輸出：fix(auth)：修正登入流程的空指標異常

範例 3：
輸入：把 React 從 17 升級到 18
輸出：chore(deps)：升級 React 至 v18
```

## 觀察重複工作的判斷流程

跑模式四的測試時：

1. 讀每個測試案例的 transcript（不只看最終輸出）
2. 注意 subagent 有沒有寫類似的 helper script
3. 如果 2 個以上的案例都獨立寫了類似功能的腳本：
   - 把那個腳本寫好，放到技能包的 scripts/
   - 在 SKILL.md 裡面告訴模型「用 scripts/xxx.py 處理這件事」
4. 這樣每次調用技能包時就不用重新發明輪子
