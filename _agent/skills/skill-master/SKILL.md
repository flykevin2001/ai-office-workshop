## 五種工作模式總覽

| 情境 | 推薦模式 | 階段 |
|------|---------|------|
| 快速做一個簡單技能包 | 模式一：引導探索型 | 建 |
| 有完整內容要整理 | 模式二：專業整理型 | 建 |
| 複雜技能包或需長期維護 | 模式三：規格驅動型 | 建 |
| 技能包做好了，要驗證品質 | 模式四：評估與優化 | 測 |
| 技能包沒被正確觸發 | 模式五：觸發調校 | 調 |
| 技能包要持續進化 | 技能包養成法 | 養 |

完整生命週期：建 → 測 → 調 → 養 → 再測 → 再調 → ...

## 技能包標準檔案結構（v3.1 統一）

所有技能包一律採用以下結構：

```
skill-name/
├── SPEC.md           ← 設計圖（規格驅動模式使用，選填）
├── SKILL.md          ← 主檔案（必要）
├── references/       ← 參考文件（按需載入）
├── scripts/          ← 可執行腳本（確定性/重複性任務）
└── assets/           ← 模板、圖示、字型等靜態資源
```

SPEC.md 不是每個技能包都需要。簡單技能包直接寫 SKILL.md 就好。複雜的、會長期迭代的才需要 SPEC.md。

## 技能包寫作指引（v3.1 新增）

這一節定義「怎麼寫出好的技能包」，適用於所有模式產出的技能包。

### Progressive Disclosure：三層載入

技能包用三層式載入，控制 context 佔用：

| 層級 | 內容 | 何時載入 | 長度建議 |
|------|------|---------|---------|
| 第一層 | name + description | 永遠在 context 裡 | 約 100 字 |
| 第二層 | SKILL.md 本體 | 技能包被觸發時 | 500 行以內（理想值，可超過但要有意識控制） |
| 第三層 | references/ 裡的文件 | 需要時才讀 | 不限，但單檔超過 300 行建議加目錄 |

設計原則：SKILL.md 放「什麼時候做什麼」的決策邏輯，references/ 放「怎麼做」的操作細節。使用者不需要每次都讀完所有 references/，SKILL.md 裡面要寫清楚「什麼時候去讀哪個檔案」。

按領域拆分 references/ 的範例：

```
cloud-deploy/
├── SKILL.md（流程 + 選擇邏輯）
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```

Claude 只讀跟當前任務相關的那一個。

### 自由度設計：不要每次都寫到一樣細

技能包不是越細越好，而是要根據任務的脆弱程度，決定要給模型多大自由度。

三種自由度：

1. 高自由度：用文字原則引導。適合情境差異大、沒有唯一正解、需要現場判斷的任務。
2. 中自由度：用偽代碼、模板、可調參數腳本。適合有偏好做法，但仍需依情境調整的任務。
3. 低自由度：用固定腳本、固定步驟、少量參數。適合容易出錯、順序不能亂、穩定性優先的任務。

判斷原則：
- 任務像走平地，路很多都能到 → 高自由度
- 任務像有欄杆的山路，需要方向但不是唯一走法 → 中自由度
- 任務像窄橋，走錯一步就掉下去 → 低自由度

寫技能包前先問自己：這件事最怕的是「做不出來」，還是「做錯一步整個壞掉」？越怕出錯，越要收斂自由度。

### 先規劃可重用資源，再寫 SKILL.md

不要一開始就衝去寫 SKILL.md。先看 2-3 個真實案例，判斷哪些東西值得抽成可重用資源。

最小資源規劃表：

| 資源類型 | 什麼時候該做 | 例子 |
|---------|------------|------|
| scripts/ | 同樣的程式碼會重寫，或任務需要高確定性 | rotate_pdf.py、clean_csv.py |
| references/ | 細節很多，但不是每次都要載入 | schema.md、api-notes.md |
| assets/ | 最終輸出會直接用到的靜態檔案 | ppt 模板、logo、前端骨架 |

原則：先列出要放什麼，再回頭寫 SKILL.md 的決策邏輯。這樣 SKILL.md 才不會變成什麼都塞的倉庫。

### Explain the Why：寫法哲學

現在的大語言模型很聰明。與其堆一堆 MUST / NEVER / ALWAYS，不如解釋「為什麼要這樣做」。模型理解了原因，就能在沒有明確規則的情境下自己判斷。

錯誤寫法：
```
MUST use exactly 3 bullet points.
NEVER exceed 200 words.
ALWAYS start with a greeting.
```

正確寫法：
```
用 3 個要點就好，因為使用者通常在手機上看，太多會滑不完。
控制在 200 字以內，因為這是給忙碌的主管看的，他只有 30 秒。
開頭先打招呼，因為這是寄給客戶的信，語氣要親切。
```

如果你發現自己在寫全大寫的 MUST / NEVER，先停下來問：「我能不能用一句話解釋為什麼？」能解釋就用解釋取代命令。

### 從回饋中歸納，不要針對個案修補

跑完 eval 之後要改技能包，改的方向要通用化。

錯誤做法：測試案例 A 的輸出缺了標題 → 加一條「輸出必須包含標題」

正確做法：測試案例 A 的輸出缺了標題 → 思考「為什麼模型會跳過標題」→ 發現是指令沒有解釋輸出結構的用途 → 補上「輸出結構的設計邏輯」，讓模型理解每個區塊的用途

差別：前者只修了一個症狀，後者讓模型理解了原則，以後遇到類似問題自己會判斷。

### 觀察重複工作，抽成腳本

跑測試的時候，如果每個測試案例的 subagent 都獨立寫了類似的 helper script（例如每次都寫一個 create_docx.py），那就該把這個腳本寫好放進 scripts/，讓技能包直接調用。

判斷標準：同樣的腳本在 2 個以上的測試案例中出現 → 值得抽出來。

### scripts 不是寫了就算，要實跑驗證

只要技能包新增或修改了 scripts/，就要真的跑一次代表性案例。目的不是把測試做得很大，而是確認這個腳本在真實環境裡能動，輸出也合理。

最低標準：
1. 至少跑一個代表性案例
2. 確認沒有語法錯誤、路徑錯誤、權限錯誤
3. 確認輸出格式符合技能包後續流程會吃的格式

如果同類型腳本很多，不必每一支都完整重跑，但至少抽樣驗證到足以建立信心。

### 輸出格式的定義方式

用範例比用規則有效：

```markdown
## 報告結構
一律用這個模板：

# [標題]
## 重點摘要
## 關鍵發現
## 建議行動
```

加上具體範例更好：

```markdown
輸入：新增了 JWT 驗證功能
輸出：feat(auth)：實作 JWT 驗證機制
```

### SKILL.md 必要結構

```markdown
---
name: 技能包ID（英文小寫加連字號）
description: 觸發說明（稍微積極一點，寧可多觸發不要漏觸發）
---

# 技能包名稱

## 核心理念（為什麼需要這個技能包）
## 核心能力（這個技能包做什麼）
## 工作流程（怎麼使用）
## 特殊處理（例外情況）
## 使用範例（實際案例）
## 重要提醒（注意事項）
## 參考文件（如有，標明什麼時候該去讀哪一個）
```

## 所有模式共通的最小流程

不管你用模式一、二、三，底層都盡量走這個順序。這是技能包大師的共通 SOP。

1. 先理解真實案例：確認使用情境，不要只抓抽象需求
2. 規劃可重用資源：決定哪些要進 scripts / references / assets
3. 初始化結構：建立資料夾與基本骨架
4. 寫 SKILL.md 與資源：先做可重用內容，再補決策邏輯
5. 快速驗證：至少檢查 frontmatter、命名、必要檔案是否正確
6. 用真實任務迭代：真的拿去跑，再根據回饋修

簡單技能包可以很快走完；複雜技能包則在每一步停久一點。但順序不要亂。

## 模式一：引導探索型（對話式開發）

### 觸發條件
- 使用者說「做技能包」「創建技能包」
- 沒有提供規格文件或完整內容

### 工作流程

第一步：需求確認。問使用者四個問題：
1. 這個技能包要解決什麼問題？
2. 使用者是誰？什麼情況下會用到？
3. 你希望它做到什麼？（核心能力）
4. 有沒有類似的技能包可參考？

第二步：資源規劃。先根據回答判斷有沒有需要提早準備 scripts/、references/、assets/。

第三步：結構建議。根據回答提出技能包名稱、觸發條件、核心能力、工作流程的初步結構。

第四步：迭代確認。使用者確認或調整後，產出完整技能包。

第五步（建議）：產出後提議使用模式四跑 2-3 個測試案例驗證。

## 模式二：專業整理型（內容轉換）

### 觸發條件
- 使用者說「把這個內容做成技能包」
- 直接提供文章、教學、方法論等完整內容

### 工作流程

第一步：內容分析。識別核心概念、適用情境、執行步驟、案例與 FAQ。

第二步：資源規劃。從內容中判斷哪些應拆到 references/，哪些需要 scripts/ 或 assets/。

第三步：結構轉換。將線性內容轉換成技能包結構：提取觸發條件、建立工作流程、設計檢查點。

第四步：品質檢查。確認邏輯完整、可操作、格式符合規範。

## 模式三：規格驅動型

### SPEC.md 的定位與用途（v3.1 重新定義）

SPEC.md 是技能包的設計圖。它記錄三件事：
1. 設計意圖 — 為什麼要這樣做（改版時回來對照）
2. 邊界定義 — 做什麼、不做什麼（防止功能漂移）
3. 驗收基準 — 怎麼算成功（跟模式四的 eval 對接）

流程：先寫 SPEC.md → 使用者確認 → 根據 SPEC.md 產出 SKILL.md

產出時嚴格照規格，不腦補、不發揮。規格沒說的就不做。

### 應用情境偏離提醒（v3.1 新增）

使用技能包的過程中，如果發現實際應用情境偏離了 SPEC.md 的設定，技能包大師要主動提醒使用者判斷：

情況 A：搞錯了 — 使用方式不對，回到 SPEC.md 原本的設計。
情況 B：情況變了 — 真實需求已經改變，SPEC.md 需要更新規格。

判斷方法：問使用者「你現在遇到的情境，跟當初設計的時候一樣嗎？」如果不一樣，追問「是你用錯了，還是情況真的變了？」

這個提醒確保 SPEC.md 不會變成過時的死文件，也不會讓使用者把技能包用到錯的地方。

### 觸發條件
- 使用者說「用規格驅動模式」「我先給你規格」
- 提供符合「技能包規格模板」的 Markdown
- 說明這是複雜技能包需要長期維護

### 工作流程

第一步：提供或協助撰寫規格。

情況 A：使用者已有規格 → 檢查完整性，確認是否符合模板格式，指出缺失部分。
情況 B：使用者需要協助 → 提供規格模板（references/skill-spec-template.md），引導填寫。

第二步：規格完整性檢查。

必須包含：核心定位、邊界設定、觸發條件、核心能力（3-5 項）、工作流程、驗收標準。
選填：特殊處理、輸出格式要求、參考資源。

第三步：先做資源規劃。

把規格中的內容拆成三類：SKILL.md 核心決策、references/ 細節說明、scripts/ 或 assets/ 的可重用資源。規格沒提到的資源不要自行補。

第四步：根據規格產出技能包。

執行原則：規格優先（不自行補充）、完整性保證、一致性保證、邊界嚴守。

產出內容：SKILL.md + references/（如規格中有提及）+ 版本號。

第五步：迭代與更新。

需求改變時：使用者修改 SPEC.md → 明確指出改了哪些 → 重新產出 → 未修改部分保持不變。

版本管理：小改 v2.2 → v2.3，中改 v2.2 → v2.5，大改 v2.2 → v3.0。

## 模式四：評估與優化

技能包做完不代表做好。評估系統讓你知道技能包「到底好不好」，而且有數據可以追蹤。

整合 Anthropic 官方 skill-creator 的評估系統，涵蓋：eval 測試、benchmark 基準測試、迭代改進。

### 觸發條件
- 使用者說「測試技能包」「跑 eval」「跑基準測試」「benchmark」
- 使用者說「技能包品質怎麼樣」「模型更新了，技能包還有用嗎」
- 技能包剛建好，進入測試階段

### 為什麼需要評估？

兩個截然相反的理由：
1. 抓退化 — 模型更新後，技能包可能反而用得更差。eval 是早期警報。
2. 發現成長 — 模型更新後，可能不需要技能包就能做到一樣好。這時候可以退役，省 token 省時間。

### 工作流程

詳細操作步驟見：references/eval-guide.md

簡要流程：

第一步：設計 2-3 個模擬真實使用情境的測試案例，存到 evals/evals.json。測試案例要像真實使用者會打的那種話，有細節、有情境、有個人脈絡，不要抽象的一句話。

第二步：每個案例跑兩版。with_skill（有技能包）和 baseline（沒技能包或舊版）。環境支援就平行跑，同一輪一起發出去。

第三步：趁測試在跑，設計量化的 assertion。assertion 要有辨別力（不能有技能包和沒技能包都通過）。主觀的東西不要硬塞量化指標，靠人工檢視。

第四步：評分、彙整 benchmark、分析數據模式、產生 HTML 檢視器給使用者看。在 Cowork 環境用 --static 產出 HTML。

第五步：讀使用者回饋，改進技能包。改的方向要通用化（參考寫作指引的「從回饋中歸納」）。改完重新跑一輪。

停止條件：使用者滿意、回饋全空、連續兩次沒實質進步。

### Forward-testing 防汙染原則

如果技能包比較複雜，或你懷疑 eval 還不夠，就做 forward-testing。但測試方式要乾淨，不然結果會失真。

原則：
1. 用新的 thread 或新的 subagent 做獨立測試，不沿用剛剛那串對話
2. 給原始任務、原始素材、原始檔案，不要把你的診斷或預期答案一起餵進去
3. 不要提示「這是在測技能包」或「這次要抓哪個 bug」
4. 每改一輪就重新從原始素材建立 context，不要沿用上一輪殘留脈絡
5. 測完後檢查並清掉容易污染下一輪的測試產物

判斷標準：如果測試成功是因為 agent 偷看了你先前的推論，那不算真的成功。

### 進階：盲測比較

把兩個版本的輸出交給獨立 agent 評判，不告訴它哪個是哪個。需要 subagent 支援。詳見官方 skill-creator 的 agents/comparator.md 和 agents/analyzer.md。

### 環境差異

| 環境 | subagent | 瀏覽器 | 建議做法 |
|------|----------|--------|---------|
| Claude Code | 有 | 有 | 完整流程，平行跑測試，開 HTML 檢視器 |
| Cowork | 有 | 無 | 完整流程，用 --static 產出 HTML |
| Claude.ai | 無 | 無 | 簡化流程，逐一跑測試，在對話中展示結果，跳過 benchmark |

## 模式五：觸發調校

技能包再好，沒被叫到就等於不存在。

### 觸發條件
- 使用者說「調校觸發」「技能包沒被叫到」「觸發不準」「優化 description」
- 技能包在模式四的測試中觸發不穩定

### 觸發機制運作原理

AI 拿使用者的提示詞跟所有技能包的 name + description 比對。注意：很簡單的任務，AI 判斷自己就能做好，可能不觸發任何技能包。目前 AI 傾向「少觸發」，所以 description 寫稍微積極一點是好的。

### 工作流程

詳細操作步驟見：references/trigger-tuning-guide.md

簡要流程：

第一步：準備 20 個測試查詢（8-10 個應觸發 + 8-10 個不應觸發）。不應觸發的要選「差一點就該觸發」的近似情境，不要選明顯無關的。

第二步：用 HTML 模板讓使用者檢視和調整。

第三步：跑優化迴圈（run_loop.py），分 train/test set 避免過擬合，最多 5 輪。

第四步：把最佳 description 更新到 YAML，展示前後對比和分數。

| 環境 | 可否跑自動化迴圈 | 替代方案 |
|------|----------------|---------|
| Claude Code | 可以 | 完整流程 |
| Cowork | 可以 | 完整流程 |
| Claude.ai | 不可以 | 手動分析弱點，人工改寫 description |

## 技能包養成法

### 核心原則：技能包是養出來的，不是塞出來的

每一次新增的內容，都必須來自實際使用中遇到的真實問題。

錯誤做法：在網路上看到好框架就塞進技能包。結果一大堆東西，不知道什麼時候該用哪個。這跟大語言模型有什麼差別？

正確做法：使用技能包時發現某個情境搞不定 → 找資料解決 → 把「問題 + 解法 + 怎麼用」一起加進去。

差別在「你知道怎麼用」。技能包存的不是知識本身，而是「我因為什麼問題，找了什麼知識，怎麼用這個知識解決了問題」。

### 技能包的兩種類型

養之前先分清楚類型，因為養法不同。

能力提升型（capability uplift）：補強模型做不好的事。有保鮮期，模型進步後可能過時。定期用模式四跑 eval 確認。

偏好編碼型（encoded preference）：記錄個人工作流程和偏好。耐久性高，模型不可能預先學會你的流程。

判斷方法：「如果有一個很強的新模型出來，它不用任何提示詞就能做到嗎？」可能可以 → 能力提升型。不可能 → 偏好編碼型。

### 四個生命階段

第一階段：誕生 — 解決一個具體問題，可能只是一個提示詞。

第二階段：成長 — 使用時發現不夠就補。觸發條件是「實際搞不定」，不是「覺得不錯就加」。比較大的新增，用模式四跑一次 eval 確認沒有改 A 壞了 B。

第三階段：裂變 — 某個子功能值得獨立。三個條件全部符合才獨立：佔比越來越大、適用情境比母技能包更廣、需要自己獨立迭代。

第四階段：通用化 — 從個人版變成可分享版。拿掉個人化的部分，留下框架和通用案例。完整版是付費產品的深度，通用版是免費體驗的入口。

### 技能包整合規則

從多個現有技能包拉內容整合時：

| 規則 | 拉進來 | 不拉 |
|------|--------|------|
| 概念層通用的原則 | 改寫成新情境的版本 | |
| 綁定特定情境的執行步驟 | | 留在原技能包 |
| 流程順序在這個技能包之後的 | | 不屬於這個技能包的範圍 |

### 養成法的自我檢查

加東西之前：這是實際問題還是「覺得不錯」？我知道怎麼用嗎？跟現有內容是什麼關係？

技能包變大時：有沒有子功能佔比太大？適用範圍比原技能包更廣？需要獨立迭代？

用了一段時間後：跑一次 eval 確認還有效。是能力提升型還是偏好編碼型？能力提升型要特別注意有沒有過期。

## 技能包格式規範

### 版本號規則
- 格式：v主版本.次版本
- 主版本：核心邏輯改變
- 次版本：功能新增或優化

### description 撰寫原則

- 說清楚做什麼
- 列出觸發條件（使用者會說什麼）
- 稍微積極：「即使使用者沒有明確說 XXX，只要情境涉及 YYY，就該使用」
- 有容易混淆的技能包就主動區分
- 目前 AI 傾向少觸發，不怕寫推一點

### 中文化處理原則
1. 全形標點：，。：；？！「」
2. 避免英文語法：不用 "not A but B"
3. 口語化表達：可用「嘛」「啦」「對」
4. 禁止粗體：不使用粗體語法
5. 禁止破折號：不使用 — 或 --

## 輸出與打包

### 創建模式
產出：SKILL.md + references/（如有）+ SPEC.md（如用規格驅動）

### 更新模式
流程：讀取現有版本 → 確認改動範圍 → 檢查 agents/openai.yaml 是否存在 → 更新版本號 → 產出新版本

如果技能包有 agents/openai.yaml，更新後必須檢查它和 SKILL.md 是否一致，特別是：
- display_name 有沒有還對應目前定位
- short_description 有沒有跟新版 description 脫鉤
- default_prompt 有沒有還符合新版工作流程

原則：UI metadata 不是附屬品，它會影響技能包在介面中的理解與使用方式。

### 打包模式

```bash
# 打包成 .skill 檔
python <skill-creator-path>/scripts/package_skill.py <技能包路徑>

# 快速驗證
python <skill-creator-path>/scripts/quick_validate.py <技能包路徑>
```

### 本地端版本管理（Obsidian 知識庫）

技能包在 `05 我的技能包/技能包名稱/` 底下管理：
- 歷史版本單檔備份：`YYYY-MM-DD-HHMM 技能包名稱 vX.X.md`
- 完整技能包資料夾（分檔版）：`YYYY-MM-DD-HHMM 技能包名稱 vX.X/`
- 更新日誌：`技能包名稱 更新日誌.md`（資料夾根目錄，獨立於所有版本之外，每一代更新持續疊加）

命名一律用中文，不用英文 skill ID。
更新日誌只寫在資料夾層的獨立檔案，不寫在技能包內部（SKILL.md 或 references/）。

## 重要提醒

### 給使用者

1. 不需要每次都跑完全部流程。簡單技能包模式一/二做好就好，重要的才跑模式四、五。
2. 規格不是限制，是保護。寫清楚規格，產出才穩定。
3. 評估不是考試。eval 的目的是幫你改進，人工檢視永遠比量化指標更可靠。

### 給 Claude

1. 規格驅動：嚴格照規格產出，不腦補。
2. 應用情境偏離：發現使用者的用法偏離 SPEC.md 時，主動提醒判斷是「搞錯了」還是「情況變了」。
3. 模式四：跑測試後一定先給人看結果（generate_review.py），再自己動手改。不要跳過 baseline。
4. 模式五：description 寫積極一點。用測試集分數選最佳版本。
5. 寫技能包時：explain the why，不要堆 MUST/NEVER。從回饋中歸納通用原則，不要針對個案修補。
6. Progressive disclosure：SKILL.md 放決策邏輯，references/ 放操作細節。
