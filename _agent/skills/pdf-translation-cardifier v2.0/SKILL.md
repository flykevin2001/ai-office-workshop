# PDF 翻譯轉檔系統

版本：2.0
設計者：江江教練
建立日期：2026-02-19
最後更新：2026-04-06

v2.0 更新說明（核心思維轉變，大版號）：
- 最大改變：不再讓 AI 直接讀 PDF。改用腳本前處理（extract_pdf.py），AI 只讀腳本抽出的純文字
- 新增三種 PDF 類型自動判斷邏輯（文字型 / 掃描型 / 混合型）
- 新增漸進式測試原則（先做 1 份確認，再批次處理）
- 新增模型分工建議（不同階段用不同等級的模型省成本）
- 觸發詞補充：「PDF 轉檔」「PDF 翻譯」
- 其餘三個階段邏輯（轉檔 / 翻譯 / 卡片化）沿用 1.3 版架構

---

## 為什麼需要這套系統

詳細說明見：references/為什麼需要這套系統.md

四個核心問題：

1. PDF 轉純文字（資料清理）：去除排版規格與亂碼，盒子越簡單，干擾越少
2. 翻譯獨立階段（上下文爆炸）：搭配字典檔分章翻譯，避免對話越長品質越差
3. 卡片化（重複使用）：拆成獨立知識單位，像積木一樣需要時重新組合
4. Markdown 格式（內容邊界）：標題劃分段落邊界，人看得懂、AI 也看得懂

---

## 系統架構

```
PDF 文件
  ↓ 腳本前處理（extract_pdf.py）
  ↓ 自動判斷類型，抽出純文字 + 圖片
  ↓ 主動告知使用者類型與下一步選項
  ↓ 使用者選擇路徑
階段一：純轉檔（_純文字.md 輸出）
  ↓ 可選繼續
階段二：翻譯（AI 只讀 _純文字.md，配合字典檔）
  ↓ 可選繼續
階段三：卡片化（長文 / 簡單卡片 / Obsidian 雙向連結卡片）
```

三個階段可以獨立使用，也可以串聯。

---

## 核心原則：腳本前處理，AI 不碰原始 PDF

v2.0 最重要的改變。

AI 直接讀 PDF 的問題：PDF 是排版格式，不是資料格式。AI 讀 PDF 時消耗大量 token 在排版代碼上，內容理解反而不穩定。

腳本前處理的好處：
- token 消耗大幅降低（只讀純文字）
- 翻譯品質更穩定（不受排版干擾）
- 可批次處理（腳本跑完整個資料夾）

腳本工具放在：`12 文字清理區/_agent/extract_pdf.py`

使用者看到腳本輸出純文字是正常的，這是流程設計，不是出錯。

## 三種 PDF 類型判斷

腳本跑完後自動判斷並告知使用者：

| 類型 | 判斷條件 | 腳本做什麼 | AI 做什麼 |
|------|---------|-----------|---------|
| 文字型 | 有文字層 | 抽文字 + 抽圖片存 assets | 只讀 _純文字.md |
| 掃描型 | 無文字層，全是圖片 | 抽圖片存 assets，文字欄空白 | 逐頁看圖片辨識文字 |
| 混合型 | 部分頁有文字層，部分頁掃描 | 有文字的頁抽文字，掃描頁抽圖片 | 先讀純文字，再補掃掃描頁 |

腳本輸出時會說明：「這份 PDF 共 N 頁，其中 X 頁有文字層，Y 頁是掃描版。」

## 兩種執行環境

| 環境 | 能力 | 適用情境 |
|------|------|---------|
| Cowork 模式 | 全自動：本地檔案系統 + Python 環境 + 批次處理 + 圖片存檔 | 大量 PDF、需要自動化、長期維護的知識庫 |
| Claude 專案模式 | 半手動：使用者上傳 PDF，Claude 在對話中處理，輸出 ZIP 供下載 | 少量 PDF（1-5 份）、不需安裝任何工具 |

兩種環境均支援圖片提取。核心格式處理邏輯相同，差別在「怎麼拿到 PDF」和「怎麼輸出結果」。

## 模型分工建議

| 任務 | 建議模型 | 原因 |
|------|---------|------|
| 純文字提取 + 格式轉換（不翻譯） | Haiku | 機械性操作，不需要語言理解 |
| 文字提取 + 翻譯 | Sonnet | 翻譯需要語言理解，但不需要頂級推理 |
| 表格結構辨識 + 複雜排版處理 | Sonnet | 需要理解版面邏輯 |
| 標籤統整 / 受控詞彙分析 | Opus | 跨文件語義比對、分類邏輯判斷 |
| 多份 PDF 交叉引用分析 / MOC 生成 | Opus | 需要理解文件間的邏輯關聯 |

預設建議：用 Sonnet 跑完全部流程，基本上所有功能都能處理。

## 漸進式測試原則

第一次跑這套流程時，禁止一口氣全部跑完。

正確順序：
1. 先做 1 份，使用者完整檢查格式（標題層級、表格、圖片 OCR、圖片嵌入連結）
2. 確認沒問題後再做 3-5 份，測試不同類型 PDF 的穩定性
3. 都沒問題了才批次處理剩餘文件
4. 批次完成後隨機抽查 3-5 份確認品質

## 階段一：PDF 轉檔

### 觸發條件

使用者說「PDF 轉文字」「PDF 轉檔」「提取 PDF 內容」或上傳 PDF。

### 步驟

1. 腳本跑 extract_pdf.py（Cowork 模式自動執行；專案模式使用者上傳後 Claude 在對話環境執行）
2. 腳本自動判斷 PDF 類型，回報頁數與文字層狀況
3. 詢問選項：

問題一：輸出格式
- .md（預設）
- .txt
- .docx

問題二：語言轉換
- 不轉換，保持原文（預設）
- 簡體轉繁體（台灣用語）
- 英文翻成繁體中文 → 建議走階段二翻譯路徑

問題三：圖片提取
- 要（預設）：圖片存 PNG，Markdown 插入 `![[嵌入連結]]`
- 不用：圖片用文字描述代替

問題四：Obsidian 雙向連結
- 要：自動生成目錄頁 + 章節雙向連結
- 不用：不加 `[[連結]]` 語法

4. 輸出純文字 _純文字.md 到輸出資料夾

### 輸出資料夾結構

PDF 檔案名稱為 `ABC.pdf`，輸出放在 `ABC/` 資料夾底下。

標準模式（頁數少）：
```
ABC/
  ABC.md
  ABC_p001_img001.png
  ABC_p005_img001.png
```

章節子資料夾模式（超過 50 頁或翻譯模式）：
```
ABC/
  ABC_MOC.md
  第一章/
    第一章.md
    第一章_p003_img001.png
  第二章/
    第二章.md
```

圖片類型判斷：
| 圖片類型 | 處理方式 |
|----------|----------|
| 照片、插圖、示意圖 | 存 PNG，插入 `![[圖片]]` |
| 表格（純文字型） | 轉成 Markdown table，不存圖片 |
| 數據圖表 | 存 PNG + 文字描述座標軸與結論 |
| 流程圖 | 存 PNG + OCR 提取圖中文字 |
| 純裝飾性圖片 | 跳過，不存 |
| 頁首頁尾 Logo | 跳過，不存 |

## 階段二：翻譯（外文文件適用）

### 觸發條件

使用者說「翻譯這份文件」「PDF 翻譯」「幫我翻譯」，或階段一完成後偵測到外文，主動詢問。

### 重要：翻譯時的讀取範圍

只讀取：
- `_純文字.md`（腳本抽出的純文字）
- 術語字典檔（若有）

不讀取原始 PDF。

### 版權確認

翻譯前確認文件來源（任一條件）：
- 合法購買的書籍或電子書
- 開放授權資料（CC 授權、公共領域）
- 供個人研究、學習使用

確認後說：「文件來源確認，請開始」

### 台灣預設字典

標點符號：全形，，。：；？！「」『』
用語原則：繁體中文，台灣用語，口語自然，避免學術腔
翻譯原則：逐句 1:1 對應，完整保留原文每一句，術語全文統一

避免的用語：
| 不要 | 要 |
|------|-----|
| 吾人應當 | 我們應該 |
| 此乃 | 這是 |
| 進行思考 | 思考 |
| 無需 | 不用、不需要 |

### 語言可調整項目

| 項目 | 預設 | 可調整為 |
|------|------|----------|
| 目標語言 | 繁體中文 | 簡體中文 |
| 地區用語 | 台灣 | 香港、中國大陸 |
| 術語保留 | 首次中英對照，之後中文 | 保留更多原文術語 |
| 語調風格 | 依照原文風格 | 口語自然、偏正式、偏學術 |

### 翻譯字典檔流程

1. 掃描文件，識別 10-20 個核心術語
2. 提出翻譯建議
3. 跟你確認每個術語翻譯
4. 生成字典檔（.md 格式）
5. 後續翻譯嚴格對照字典檔
6. 翻譯過程中遇到新術語 → 暫停 → 建議 → 確認後加入 → 繼續

### 翻譯進度管理

每段完成後提示：
```
已完成：[章節名稱]
進度：X / Y 章
下一章：[下一章名稱]
繼續嗎？
```

完成一個大章節時建議開新視窗繼續（避免對話過長影響品質）。

## 階段三：卡片化

### 觸發條件

使用者說「卡片化」「轉成 Obsidian」「幫我做卡片」，或翻譯完成後主動詢問。

### 三種模式

我會問：「要哪種卡片化模式？」

模式 A：長文模式
全部內容輸出成一篇長 Markdown 文件。適合只要完整翻譯文件。

模式 B：簡單卡片模式
每章一張卡片，獨立檔案，不加 Obsidian 雙向連結。適合分章節管理但不用 Obsidian。

模式 C：Obsidian 雙向連結卡片模式
每章一張卡片，加完整雙向連結，含 MOC 目錄頁。適合使用 Obsidian 管理知識。

### Obsidian 卡片格式（模式 C）

MOC 目錄頁：
```markdown
# 書名｜MOC

tags: #書名 #MOC #目錄

## Part I：[大章節名]
1. [[書名｜章節名稱一]]
2. [[書名｜章節名稱二]]
```

章節卡片：
```markdown
# 書名｜章節名稱

tags: #書名 #[相關標籤]

延續：[[書名｜上一章名稱]]

[章節內容]

接下來請看：[[書名｜下一章名稱]]
回到目錄：[[書名｜MOC]]
```

注意：第一章不加「延續」行，最後一章不加「接下來請看」行。

### 檔名分隔符號

預設使用全形直線 ｜。選項：半形連字號 -、全形冒號 ：、底線 _

### 檔名與標題一致規則

每張卡片的檔名必須與文件第一行標題（# 開頭那行）完全一致。
範例：
- 檔名：`納瓦爾寶典｜理解財富創造.md`
- 文件第一行：`# 納瓦爾寶典｜理解財富創造`

---

## 標籤規劃系統

### 執行時機

進入卡片化階段時，開始產出卡片之前，先做一次標籤規劃。

### 流程

1. 快速掃章節結構和主題，提出 3-5 個核心標籤建議
2. 確認標籤形式：一般標籤 `#標籤` 或連結卡片標籤 `[[標籤]]`
3. 確認標記系統（來源 / 狀態 / 專案標記）
4. 建議是否建立受控詞彙表

標籤數量原則：每張卡片 3-5 個標籤，全書核心標籤控制在 3-5 個，總標籤數不超過 8 個。

---

## 防呆機制

- 沒有字典檔就開始翻譯 → 建議先建立術語對照表
- 沒有標籤規劃就開始卡片化 → 先快速規劃再開始
- 新對話續讀 → 告知從哪章開始 + 確認規格檔在知識庫中
- 對話過長 → 完成大章節後提醒換新視窗繼續
- 圖表處理 → 四步驟：OCR 文字 → 描述視覺結構 → 解釋核心概念 → 用【圖表描述】標記

## 限制說明

- 複雜排版（多欄、文繞圖）可能無法完美還原
- 手寫字 OCR 準確度較低
- 加密 PDF 需要先解密
- 長文件建議分多次對話處理（每次一個大章節）
