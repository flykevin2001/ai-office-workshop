# 第二階段：Markdown → Obsidian 知識圖譜

第一階段轉出 Markdown 後，可選擇接入知識圖譜整理。

---

## 觸發方式

- 使用者說「轉知識圖譜」「整理成筆記」「拆成 Obsidian」
- 或在模式一轉完後，使用者說「繼續整理」「幫我拆」

---

## 兩種深度

### 輕量版（預設）

一份文件 = 一個 .md 筆記

產出：
1. 一個 .md 檔案，保留完整內容
2. 標題改成知識庫格式
3. 加上 `[[]]` 雙向連結（指向知識庫裡已有的相關筆記）
4. 上標籤（走四維度標籤流程）
5. 存到知識庫根目錄

什麼時候用：
- 單一主題的文件（一份報告、一篇文章）
- 只想收進知識庫，不需要深度拆解
- 趕時間

### 完整版（使用者說「完整版」「拆深一點」時才做）

一份文件 → 多個 .md 筆記 + MOC 索引頁

產出：
1. **MOC 索引頁**：`YYYY-MM-DD-HHMM MOC：主題名稱.md`
   - 列出所有子筆記的連結
   - 簡短說明每個子筆記的重點
2. **子筆記**：每個章節 / 主題一個 .md
   - 標題格式：`YYYY-MM-DD-HHMM 主題名稱：子主題.md`
   - 每個子筆記都有 `[[]]` 連回 MOC
   - 子筆記之間也建交叉連結
3. **四維度標籤**：每個子筆記獨立上標籤
4. 全部存到知識庫根目錄（或使用者指定的資料夾）

什麼時候用：
- 多章節的長文件（書籍、大型報告）
- 需要跨章節交叉引用
- 想深度整理成知識網絡

---

## 拆分策略判斷

AI 讀完 Markdown 後，判斷適合哪種深度：

| 條件 | 建議 |
|------|------|
| 內容 < 2000 字 | 輕量版 |
| 內容 > 2000 字，但單一主題 | 輕量版 |
| 內容 > 2000 字，有明確章節 | 完整版（但先問使用者） |
| 使用者說「拆深一點」 | 完整版 |
| 使用者沒特別說 | 輕量版 |

---

## 標籤流程

走既有四維度標籤流程（不重新發明）：

1. 讀取 `tag-dictionary.md`
2. 依 `tag-rules.md` 七步驟判斷
3. 字典沒有的 → 問使用者要不要新增
4. 寫入標籤
5. 問使用者確認

---

## 連結邏輯

複用 `admin-assistant/references/web-to-knowledge-graph.md` 的連結規則：

1. **內部連結**：文中提到的人名、工具名、概念名，如果知識庫裡有對應筆記，加 `[[]]`
2. **外部連結**：保留原始 URL，不轉成內部連結
3. **MOC 連結**（完整版）：每個子筆記連回 MOC，MOC 連到所有子筆記
4. **交叉連結**（完整版）：子筆記之間如果有關聯，互相連結
